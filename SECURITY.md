# 安全策略

本文件定义本仓库的安全漏洞报告流程、响应时间承诺、披露窗口与适用范围。任何使用者发现潜在安全问题，请按本流程上报，切勿直接公开披露或在 Issue 中描述漏洞细节。

## 报告漏洞

若你发现安全漏洞或潜在的安全风险，请通过以下方式私密上报，**不要在公开 Issue 列表中创建漏洞相关 Issue**，也不要在 PR、讨论区或社交平台公开漏洞细节：

- 优先邮件：fanquanpangpiing@163.com
- 邮件主题请使用 `[SECURITY] <仓库名> - <简短问题描述>` 格式
- 邮件正文请尽量包含以下信息：
  - 受影响的仓库与分支版本（默认聚焦 `main` 分支最新代码）
  - 漏洞复现步骤
  - 影响范围与可能的攻击场景
  - 建议的修复方向（可选）
  - 报告者希望使用的署名信息（用于后续致谢，可匿名）

收到报告后，维护者将通过同一邮箱与报告者沟通后续进展，避免漏洞细节外泄。

## 响应时间

维护者按以下时间窗口响应安全报告：

- **48 小时内**：确认收到报告，并向报告者回执。
- **7 天内**：完成初步评估，给出是否认定为安全漏洞、影响等级（严重 / 高 / 中 / 低）与初步处置计划。
- **30 天内**：对于被认定为有效漏洞的报告，给出修复方案或临时缓解措施。
- **90 天内**：完成修复并通过协调披露流程公开漏洞信息与修复致谢。

如遇节假日或特殊情况导致响应延迟，维护者将在以上时间窗口内向报告者主动说明延迟原因与预计响应时间。

## 披露窗口

本仓库采用 **90 天协调披露**（Coordinated Disclosure）策略：

- 报告者提交有效漏洞后，维护者在 90 天内完成修复并发布修复版本。
- 在漏洞修复版本发布前，维护者与报告者共同遵守披露静默期，不对外公开漏洞细节、复现步骤与受影响代码路径。
- 修复版本发布后，维护者将通过 Release Notes 或安全公告公开漏洞概述、影响范围与修复版本信息，并对报告者予以致谢（除非报告者要求匿名）。
- 在以下情形下，披露窗口可由双方协商调整：
  - 漏洞影响范围极为广泛，需要更长修复周期；
  - 报告者主动同意延长静默期；
  - 漏洞已被第三方公开披露，无可继续协调的窗口。

## 适用范围

本安全策略仅适用于以下范围：

- 本仓库 `main` 分支上的最新代码。
- 本仓库官方发布的 Release 版本中已知的受影响代码路径。
- 本仓库默认配置下运行的行为；若使用者自行修改配置、关闭安全选项或对接未授权的第三方依赖所引发的安全问题，不在本策略覆盖范围。

对于历史分支、已被废弃的实验性代码或仓库历史归档版本中的漏洞，维护者将视严重程度决定是否处置，不强制纳入响应时间承诺。

## 不在范围内的行为

以下行为不属于本仓库鼓励的安全测试范围，相关报告将不予受理，必要时维护者将保留采取进一步措施的权利：

- 对本仓库线上服务进行分布式拒绝服务（DDoS / DoS）攻击。
- 对本仓库维护者、贡献者或用户进行社会工程学攻击、钓鱼攻击或身份冒充。
- 针对本仓库物理基础设施、办公网络或个人设备的物理安全测试。
- 大规模自动化扫描、爆破、 fuzzing 给线上服务造成明显负担的行为。
- 已知未修补的第三方依赖 CVE 漏洞，且本仓库已在依赖配置中提供最新可用版本，但使用者未及时升级所致的问题。
- 利用本仓库代码在非授权环境部署后，因部署配置不当引发的安全问题。
- 在漏洞未修复前，公开披露漏洞细节或对外提供可复现的 PoC。

## 致谢

对于提交有效安全漏洞的报告者，本仓库将在漏洞修复公告中予以致谢：

- 默认使用报告者邮件中提供的署名信息（GitHub 用户名、姓名或昵称）。
- 若报告者要求匿名，将仅以 `匿名报告者` 字样致谢，不公开任何个人信息。
- 致谢信息将随修复版本 Release Notes 一并发布，发布后不再撤回，除非报告者提出明确且合理的撤回请求。

维护者感谢每一位负责任披露安全问题的贡献者，并承诺在沟通过程中保持专业、尊重与透明。
